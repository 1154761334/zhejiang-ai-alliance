# 浙江省 AI 智能体产业联盟平台 - 全流程验收测试计划 (V2.0)

## 1. 测试目标
建立符合软件工程标准的闭环验收体系，确保系统不仅“功能可用”，而且“生产可靠”且“流程合理”。

## 2. 测试阶段划分

### 第一阶段：后端接口与业务逻辑层 (API Validation)
**重点**：验证数据的完整性、安全隔离性和计算逻辑的准确性。
- **工具**：`node backend/scripts/test-workflow-api.mjs` (扩展脚本)
- **核心用例**：
  - [ ] **鉴权与隔离**：验证非 ADMIN 角色无法调用 `getAllCompaniesForExport` 及修改他人账号。
  - [ ] **密码更新逻辑**：验证新旧密码校验及 Directus 用户状态更新。
  - [ ] **评估引擎准确性**：针对不同偏差案例（如人数水分 > 50%）验证 `lib/evaluation.ts` 的输出等级。
  - [ ] **CSV 数据流**：验证 API 返回的扁平化数据是否包含所有 Tier A/B/C 的必填项。

### 第二阶段：前端交互与易用性 (UI/UX Testing)
**重点**：验证用户操作的连贯性、容错处理和视觉反馈。
- **核心环节**：
  - [ ] **响应式表单**：在不同分辨率下测试企业入库申请流程。
  - [ ] **CRM 操作流**：侧边栏 CRM 操作的实时同步和保存提示。
  - [ ] **数据包下载**：验证生成的走访资料包（CSV）在 Excel/WPS 中的编码兼容性（UTF-8 BOM）。

### 第三阶段：产品经理视角审计 (PM Review & Flow Audit)
**重点**：跳出代码，从联盟秘书处和企业用户的实际生产场景出发。
- **审查维度**：
  - **流程冗余**：是否存在不必要的层级？例如，B-Tier 和 C-Tier 的录入是否过于重叠？
  - **表单合理性**：字段命名是否过于技术化？是否符合政企对接的习惯（如“Revenue Range”改为“年营收规模”）。
  - **功能缺失/过度设计**：目前的评分系统是否对秘书处决策起到了“降噪”作用，还是增加了录入负担？

## 3. 验收标准
1. **API 通过率**：100% (核心流全绿)。
2. **逻辑一致性**：自报数据、外部数据、尽调数据在 CSV 中无缝对齐。
3. **用户体验**：平均任务完成路经最短（例如：3次点击内完成审核建议录入）。

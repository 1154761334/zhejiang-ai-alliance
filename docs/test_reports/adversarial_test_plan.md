# 浙江省 AI 智能体产业联盟平台 - 破坏性与极限压力测试计划 (Adversarial Test Plan)

## 1. 核心指导方针：扮演“鲁莽用户”
我们要忘记开发者的“由于是我写的所以它没问题”的直觉，转而采用“如果我乱点会发生什么”的攻击性视角。

## 2. 攻击/异常测试场景

### A. 账户安全与权限越权 (Security & Privilege Escalation)
- **跨用户改密攻击**：尝试直接从前端修改 Server Action 参数，请求修改非本人 ID 的密码。
- **参数注入**：在改密接口尝试发送 `null`、超长字符（1万位）、特殊脚本字符等。
- **非法提权**：以普通企业用户身份尝试直达 `/admin` 路由及调用 `export` 接口。

### B. 极限数据录入 (Data Integrity & Edge Cases)
- **虚空尽调**：尝试保存一个全空的 Tier C 尽调记录。
- **数值溢出**：人员规模输入 `-999` 或 `999,999,999`。
- **特殊字符破坏**：在公司名称或产品描述中输入 `'; DROP TABLE companies;--` (SQL 注入测试) 或 `<script>alert('XSS')</script>` (跨站脚本测试)。

### C. 流程“乱搞” (Broken UX Flows)
- **并发提交**：在保存 A-Tier 数据时疯狂连击保存按钮（验证是否有 Loading 防重机制）。
- **断网恢复**：在提交表单中途切断 API 响应，观察前端是否有优雅的报错反馈，还是直接白屏。
- **导出极限**：在数据库有 0 条数据或有 10,000 条数据时点击导出全景内容。

### D. UI “找茬” (UI/UX Imperfections)
- **文字换行破坏**：给一个长达 100 字的产品名称，看它是否撑破了卡片布局。
- **非法导航**：在一个需要引导的 4 步表单中，强行通过 URL 跳转到第 4 步。

## 3. 工具与执行
- **Adversarial Script**：编写 `test-adversarial.mjs`。
- **Browser Monkey Test**：利用浏览器工具模拟非理性点击和输入。

## 4. 修复策略
所有发现的“异常现象”必须在代码层进行 Zod 校验加强或防抖处理。
